# 【AI Builders解读02】AI Builder 导演工具箱｜配套工具箱

> 来源：根据 Kim Beverett 在 Config 2026 的演讲 `Be the director: how stage craft informs product craft` 提炼。  
> 说明：“五秒第一印象测试”“一句话取舍模板”和“用户试用观察表”是本期解读整理出的可复用工具，不是 Kim 在原演讲中使用的正式名称。

这三张工具卡组成一个闭环：

> 一句话取舍模板决定你想坚持什么；五秒第一印象测试检查第一眼有没有传达到；用户试用观察表验证完整使用过程是否成立。

## 工具一｜五秒第一印象测试

### 来源关系

Kim 用三种椅子摆放说明：观众会从每一个舞台选择中形成预期；放到产品里，用户也会把页面上的每个位置和样式理解成 Builder 的决定。“五秒第一印象测试”是本期根据这个观点整理的检查方法。

### 什么时候使用

适合首页、新功能入口、Agent 开始执行前的确认页面，以及每次大改之后。

### 输入

- 一个准备测试的页面、截图或可点击原型；
- 一位没有参与设计的目标用户或近似用户；
- 下面三项由 Builder 预先写好的预期。

### 测试前先写

- 我希望用户第一眼看到：______
- 我希望用户觉得这个页面是做什么的：______
- 我希望用户下一步做：______

### 测试方法

找一位没有参与设计的人，不做任何解释，只给他看页面五秒，然后关掉页面，依次询问：

1. 你第一眼看到了什么？
2. 你觉得这个页面是做什么的？
3. 如果继续操作，你下一步会做什么？

### 记录结果

| 项目 | 我原本希望 | 用户实际回答 | 差距 |
| --- | --- | --- | --- |
| 第一眼注意力 |  |  |  |
| 页面用途 |  |  |  |
| 下一步操作 |  |  |  |

一次只修改一个最影响理解的元素，再进行下一轮测试。这样才能知道究竟是哪项修改产生了效果。

### 可复制提示词｜汇总多轮五秒测试

> 先完成真实用户测试，再把记录交给 AI。不要让 AI 假装成用户替代测试。

```text
你是我的用户研究整理助手。下面是多位真实用户完成“五秒第一印象测试”后的原始记录。

设计预期：
- 希望用户第一眼看到：【填写】
- 希望用户认为页面用途是：【填写】
- 希望用户下一步做：【填写】

真实用户记录：
【逐条粘贴，每条注明用户编号；不要删除原话】

请按以下规则整理：
1. 只依据记录中明确出现的内容，不模拟用户，不补充用户没有说过的原因。
2. 分别汇总“第一眼注意力、页面用途、下一步操作”三类回答。
3. 每类列出：
   - 重复出现的模式；
   - 与设计预期一致的证据；
   - 与设计预期不一致的证据；
   - 仍无法判断的问题。
4. 所有判断必须附对应的用户编号和原话摘录。
5. 给出最多 3 个值得优先检查的页面元素，并说明支持它的证据。
6. 不要替我决定最终改版方案。最后提醒我选择下一轮只修改的一项。

请使用表格输出，最后单独列出“证据不足，不能下结论”的事项。
```

### 输出

得到一处优先级最高的“预期与实际理解差距”，以及下一轮只修改的一项内容。

### 使用边界

它只能检查第一印象，不能替代完整的可用性测试。单个用户的回答也不是结论；如果准备作重要决定，应找多位目标用户重复测试，观察是否出现相同模式。

## 工具二｜一句话取舍模板

### 来源关系

Kim 借 Peter Brook 的《仲夏夜之梦》说明，一部作品需要一条能贯穿具体选择的 `spine`。她也强调，这条原则不能由 AI 替人决定。“一句话取舍模板”是本期把这个观点转化成的产品判断工具。

### 什么时候使用

适合产品刚开始确定方向、AI 同时给出多个都能用的方案、团队对功能取舍没有共同判断标准，或准备复盘现有功能时。

### 输入

- 产品背景与目标用户；
- 一个关键使用场景；
- 希望用户获得的核心结果；
- 当前互相冲突的方案、功能或价值；
- 已有的真实用户反馈。

### 最终输出格式

> 当【目标用户】处在【关键场景】时，我们宁可【选择 A】，也不要【选择 B】，因为用户真正需要的是【核心体验或结果】。

示例：

> 当 AI 没有把握时，我们宁可多问用户一次，也不要让它偷偷替用户做决定，因为用户真正需要的是可控。

### 判断它是否有用

- 是否写清楚一个具体用户和场景？
- 是否包含两个不能同时满足的选择？
- 是否明确愿意放弃什么？
- 是否能指导页面、文案和 AI 行为？
- 是否能帮助你拒绝一个看起来也不错的方案？

如果最后一个问题的答案是“不能”，它可能还只是一句口号。

### 可复制提示词

```text
你是我的产品原则教练。你的任务不是替我做最终决定，而是通过提问，帮助我找出这项产品真正需要坚持的取舍。

产品背景：【填写】
目标用户：【填写】
关键使用场景：【填写】
希望用户最终获得的结果：【填写】
目前纠结的方案或功能：【填写】
已有用户反馈：【填写】

请按照以下方式工作：
1. 每次只问我一个问题，最多问 5 个问题。
2. 优先追问：两个方案冲突时，我愿意牺牲什么，绝不能牺牲什么。
3. 不接受“简单、好用、创新、有温度”等所有产品都能使用的形容词。
4. 提问结束后，给出 5 条候选产品原则。每条都使用这个格式：
   “我们宁可 ______，也不要 ______，因为用户真正需要的是 ______。”
5. 每条候选原则下面分别列出：
   - 它会支持的两个产品决定；
   - 它会拒绝的一个合理但不合适的决定；
   - 它可能带来的代价。
6. 不要替我选择最终答案。最后请我结合自己的经验重写并确认。
```

确认最终原则后，可以继续让 AI 检查现有方案：

```text
这是我确认的产品原则：【粘贴】
这是当前的功能和方案：【粘贴】

请严格依据这条原则，将方案分为“保留、修改、删除”三组，并解释每个判断。不要新增功能，也不要修改我的产品原则。
```

### 使用边界

提示词负责追问、生成候选和检查冲突，不能替 Builder 决定最终原则。原则也不是永久口号；当目标用户、关键场景或证据发生变化时，需要重新检查。

## 工具三｜用户试用观察表

### 来源关系

Kim 讲到 Peter Brook 把真实观众请进排练，也用 Codex `app shots` 一年间反复试验的过程说明：真实体验不是一次生成出来的。“用户试用观察表”是本期据此整理的行为记录工具。

### 什么时候使用

适合关键流程已经能真实操作，但还没有正式发布；也适合上线后准备修改交互流程之前。

### 输入

- 本轮只想验证的一个问题；
- 一个能够实际操作的版本；
- 一项符合真实场景的用户任务；
- 团队原本预期用户会如何理解和操作。

### 使用步骤

1. 每轮只写下一个最重要的验证问题。
2. 给用户一个真的能够操作的版本和一项符合真实场景的任务。
3. 不先解释正确用法，也不先问“喜不喜欢”，只观察实际行为。
4. 记录下面三个“第一次”。
5. 根据证据决定保留、修改呈现方式，还是重新检查产品原则；下一轮只修改一项。

用户提供证据，Builder 负责作出最终决定。

### 三个“第一次”

1. 第一次停顿发生在哪里？
2. 第一次理解错了什么？
3. 第一次实际操作和团队预想得不一样，发生在哪里？

### 一页记录模板

```text
本轮想验证的问题：
目标用户：
交给用户的任务：
我们希望用户自然理解：

第一次停顿：
第一次理解错误：
第一次与预想不同的操作：
用户以为下一步会发生：
真正让任务变简单的地方：

本轮决定：保留 / 修改呈现方式 / 重新检查产品原则
下一轮只修改：
```

不要一次改动多个变量。每轮只改变最可能影响理解的一项，再让下一位用户重新尝试。

### 可复制提示词｜整理用户试用观察证据

```text
你是我的可用性测试记录助手。你的任务是整理真实行为证据，不是替用户解释动机，也不是替团队做产品决定。

本轮想验证的问题：【填写】
目标用户：【填写】
交给用户的任务：【填写】
团队预期的理解与操作路径：【填写】

原始观察记录：
【按时间顺序粘贴；保留用户原话、操作和时间点】

请按以下规则处理：
1. 区分三类内容：
   - 可直接观察到的行为；
   - 用户明确说出的原话；
   - 观察者的推测。
2. 找出并标记：
   - 第一次停顿；
   - 第一次理解错误；
   - 第一次与团队预想不同的操作。
3. 每条证据写清：发生时间、用户做了什么、用户说了什么、与预期的差异。
4. 不把停顿自动解释成困惑，不把一句偏好直接解释成需求。
5. 汇总最多 3 个跨用户重复出现的模式；没有重复证据时明确写“尚未形成模式”。
6. 基于证据提出下一轮可验证的问题，并给出三个选项：
   - 保留当前方案；
   - 只修改呈现方式；
   - 回到产品原则重新检查。
7. 不要替我选择。最后请我确认下一轮只修改的一项内容。

输出格式：
A. 证据时间线
B. 三个“第一次”
C. 跨用户模式
D. 仍无法判断的问题
E. 下一轮验证选项
```

### 输出

得到三个真实行为证据，以及一个明确的下一轮决定：保留、修改呈现方式，或重新检查产品原则。

### 使用边界

观察表记录的是行为，不会自动解释行为背后的原因。完成任务后可以用中性问题追问，但不要把用户的一句偏好直接当成产品决定；证据由用户提供，取舍仍由 Builder 负责。
